# v0.2.2 — Sentence basis tagging (evidence vs. inference)

Date: 2026-04-22
Scope: backend writing service + frontend LetterPanel
Status: implemented, build green, pending prod backend restart

---

## Task Understanding

In the Write Panel, every body sentence of the generated petition letter
should visually signal where its claim comes from:

- **Evidence-grounded** (the sentence restates, quotes, or paraphrases a
  snippet from the record): persistent **blue** background.
- **Inference/argumentation** (the sentence draws a legal conclusion,
  characterization, or transitional claim that goes beyond any single
  snippet): persistent **yellow** background.

The colors are always on (not focus-only). When a sentence becomes the
focused one (exhibit focus or argument/sub-argument focus), its tint
darkens.

The classification must come from the **writing LLM itself** — not only
the argument-tree structure — so the model is the one deciding each
sentence's epistemic basis. A structural fallback (snippet_ids non-empty
→ evidence) covers older data and LLM omissions.

## Implementation Plan

1. Add `basis: 'evidence' | 'inference'` to the sentence schema at both
   ends (frontend type + backend Pydantic model + on-disk JSON).
2. Extend both writing prompts (`_write_single_subargument` and
   `_step1_generate_argument_body` in `petition_writer_v3.py`) so the
   LLM returns a `basis` tag per sentence, with a precise definition of
   each label.
3. Normalize the LLM output via a new `_normalize_basis()` helper —
   accepts `"evidence"`/`"inference"` case-insensitively, falls back to
   a structural default when missing or malformed.
4. Thread `basis` through the two final-assembly sites
   (`_assemble_sentences_v3` and the per-argument orchestrator) so
   body sentences carry the tag; opening/closing framing sentences are
   hard-coded to `"inference"`.
5. Render persistent basis tint in `LetterPanel.tsx`:
   - evidence → `bg-blue-50` default, `bg-blue-100` on hover, `bg-blue-200` when focused
   - inference → `bg-yellow-50` default, `bg-yellow-100` on hover, `bg-yellow-200` when focused
   - opening/closing → no basis tint (they're framing, not claims)
6. Clean up the now-unused `hoveredSentenceIdx` React state; hover
   intensification is handled entirely by Tailwind's `hover:` utility.

## Changes Made

### Frontend
- `frontend/src/types/index.ts` — `SentenceWithProvenance.basis?: 'evidence' | 'inference'`.
- `frontend/src/components/LetterPanel.tsx` —
  - Compute `basis` per body sentence (LLM tag first, structural fallback)
  - Apply persistent + hover + focused background classes by basis
  - Drop the old hover state (`hoveredSentenceIdx`) and mouse handlers

### Backend
- `backend/app/routers/writing.py` — add `basis: Optional[str]` to
  `SentenceWithProvenanceV3`.
- `backend/app/services/petition_writer_v3.py` —
  - New `_normalize_basis(value, snippet_ids)` helper
  - Prompt updates in `_write_single_subargument` and
    `_step1_generate_argument_body` with definitions of evidence vs.
    inference + example JSON that includes the new field
  - Post-processing in both LLM paths normalizes `basis` with fallback
  - `_assemble_sentences_v3` and the top-level assembly block in the
    per-argument orchestrator now emit `basis` on every sentence
    (opening/closing forced to `"inference"`)

## Verification

- `npm run build` → 269 modules, 7.79s, no errors.
- Python smoke:
  `_normalize_basis('evidence', ['x'])` → `'evidence'`
  `_normalize_basis('INFERENCE', [])` → `'inference'`
  `_normalize_basis(None, ['x'])` → `'evidence'`
  `_normalize_basis(None, [])` → `'inference'`
  `_normalize_basis('bogus', [])` → `'inference'`
- Existing on-disk writing JSON has no `basis` field; the frontend
  falls back to snippet-based inference so old data renders correctly
  without migration.

## Risks / Pending

- **Prod backend restart required**: `petiton-backend.service` runs
  without `--reload`. New prompt + normalizer only take effect after
  `systemctl restart petiton-backend`. Deferred to user go-ahead to
  avoid an unscheduled blip on `plus.drziangchen.uk`.
- **LLM compliance**: DeepSeek may occasionally omit or miscategorize
  `basis`. The structural fallback keeps the UI coherent, but the
  tagging quality is ultimately prompt-dependent and worth eyeballing
  on the first few generated sections.
- **Polish step untouched**: `_step2_polish_single_subarg` doesn't
  rewrite `snippet_ids`, but it also doesn't touch `basis`. Polished
  sentences retain the basis set in step 1.
- **Scope not included**: span-level (sub-sentence) highlighting was
  explicitly deferred; granularity stays at sentence level, consistent
  with the existing provenance model.
- **Rollback**: `git revert` this commit; backend auto-normalizes
  legacy sentences (without `basis`) by structural fallback so data on
  disk remains forward/backward compatible.
