# v0.2.1 — Dehuan PDF 404 hotfix + Arbor branding

Date: 2026-04-22
Scope: frontend (title/icon) + backend project data (meta.json only, no code)
Status: implemented, verified via curl + production build

---

## Task Understanding

User reported two issues after shipping v0.2.0:
1. `GET plus.drziangchen.uk/api/documents/dehuan_liu/pdf/A1` → 404.
   Frontend `DocumentViewer` failed to load any Dehuan Liu exhibit PDF.
2. Browser tab shows "User Study – Evidence-First Authoring" with the
   default Vite SVG. User wants the app to present as **Arbor**.

## Root cause (issue 1)

`dehuan_liu/meta.json` carried a stale Windows `sourcePath`
(`F:\\Python-Project\\...\\eb1a\\Dehuan Liu`) from the data's original
authoring machine. `backend/app/routers/documents.py::_get_source_path`
reads this field and builds `<sourcePath>/PDF/<letter>/<exhibit>.pdf`.
On the Linux server that path does not exist, so every exhibit 404s.

Live backend reads from `PETITON_PROJECTS_DIR=/root/petiton-data/projects`
(not the repo copy); that meta.json had the same Windows path.

## Implementation Plan

1. Fix `sourcePath` in **both** the in-repo
   `backend/data/projects/dehuan_liu/meta.json` and the live
   `/root/petiton-data/projects/dehuan_liu/meta.json` to
   `/root/PetitonLetterUserstudyFrontend/backend/data/eb1a/Dehuan Liu`
   (confirmed to contain `PDF/A/a1.pdf` etc.).
2. Update `frontend/index.html`: `<title>Arbor</title>` and point the
   favicon link at a new `/favicon.svg`.
3. Add `frontend/public/favicon.svg` — simple dark-green tile with a
   light tree silhouette (matches the "Arbor" name).
4. `npm run build` to confirm; curl the PDF endpoint locally and on prod.

## Changes Made

- `backend/data/projects/dehuan_liu/meta.json` — `sourcePath` updated to

  200 (was 404).

## Risks / Pending

- **Other project data drift**: Only `dehuan_liu` was fixed. If more
  projects are added later with Windows-authored `meta.json`, the same
  issue will recur. Out of scope for this hotfix.
- **GH Pages rebuild**: Title/icon will only update once the GitHub
  Actions deploy workflow runs against the upcoming commit.
- **Rollback**: `git revert` the upcoming commit; restore the old
  `sourcePath` in the live meta.json if needed.
